# 项目级多 Agent 安全审计

- 审计日期：2026-07-24
- 范围：Codex 项目配置、自定义 Agent、命令审批、持久通信、任务状态和辅助脚本
- 结论：适合可信本地开发团队使用；不应视为对抗恶意 Agent 的强隔离系统

## 已修复

### 持久通信缺口

原始版本只依赖 Codex 原生线程回传，跨会话时阻塞和决策可能丢失。

现已增加 `coordination/` 和 `scripts/agent_bus.py`：

- 非主 Agent 直接通信自动抄送主 Agent；
- 消息和确认使用独立不可变文件；
- Agent、任务和消息 ID 白名单校验；
- 路径穿越防护；
- 32 KiB 正文上限；
- 高置信度凭据模式拒绝；
- SHA-256 内容完整性检查；
- 独立确认记录；
- 完成门阻止存在未确认关键消息的任务完成。

### 外部连接器写操作

项目配置现将连接器写操作的默认审批模式设为 `writes`，审批人固定为用户。

### 环境变量与 Shell

子进程只继承核心环境，禁用登录 Shell，并额外排除常见云凭据、密码和 Cookie 变量。

### 敏感命令规则

规则已补充 Git 覆盖/删除、软件包发布、容器推送、集群、基础设施和常见部署命令，并为每条
规则加入内联 `match` 样例。验证脚本同时检查基本语法、规则数量和必需前缀。

### 虚假完成

增加 `scripts/complete_task.py` 完成门，拒绝：

- 状态不是 `VERIFY`；
- 仍有未勾选验收项；
- 验证表没有实际记录；
- 最终结果为空；
- 仍有待用户决定的关联审批；
- 仍有未确认的关键 Agent 消息；
- 完成目录已有同名任务。

## 已验证

- 项目 TOML 可以被 Python `tomllib` 解析；
- 5 个自定义 Agent 都含必需字段；
- 安全审查 Agent 为只读；
- 其他实施 Agent 包含持久通信约束；
- 规则文件通过基本语法和结构检查；
- F 盘支持消息脚本使用的硬链接原子发布；
- 9 项单元及负向测试通过；
- 工作区综合验证通过。

## 剩余边界

### 会话权限覆盖项目默认值

Codex 会把父会话的实时权限选择应用给子 Agent。用户如果选择完全访问或类似的宽松模式，
可能覆盖项目或自定义 Agent 的默认设置。仓库文件无法强制撤销这一运行时选择。

### 共享工作区不是敌对隔离

`workspace-write` 允许写当前工作区。前端、后端和测试的目录所有权是治理规则，不是不同
操作系统账户或 ACL。具有写权限的恶意进程可以越界修改文件，也可以伪造消息身份并重新生成
内容哈希。主 Agent 必须检查实际 diff 和原生线程来源。

### 命令规则不是完整安全边界

Codex 项目规则控制申请在沙箱外运行的匹配命令，并且该功能仍可能演进。它不能替代沙箱、
连接器审批和人工确认。当前环境中的 Codex App 可执行文件受 WindowsApps ACL 限制，未能直接
运行官方 `codex execpolicy check`；项目验证器只做语法、结构和样例覆盖检查。

### 持久消息箱不自动唤醒 Agent

消息箱是审计与恢复机制。实时发送、跟进、等待和停止 Agent 仍由 Codex 原生子 Agent 功能承担。

### 源码路径与技术栈需要在目标项目绑定

母版不包含前端或后端业务占位目录。复制到目标项目后，必须在根 `AGENTS.md` 和对应 Agent
配置中登记真实源码路径，并补充安装、构建、lint、类型检查、测试和安全扫描命令，
之后才能对应用代码做端到端验收。
